What is Return Oriented Programming

Return Oriented Programming (ROP) was discovered in 1997 by Solar Designer. Its main concept is to reuse existing code within the binary itself to hijack control flow during process execution.

The generalization of Return-to-libc is Return Oriented Programming. It is commonly used to bypass modern exploit mitigations like Address Space Layout Randomization (ASLR) and the Non-Executable (NX/DEP) bit.

1. Resources on the Stack

The stack acts as the attacker's "scratchpad". When you overwrite a buffer and take control of the saved return address, the stack will contain:

  • Return addresses (pointers to the specific gadgets you want to run)
  • Data (constants, strings, syscall numbers, etc.)
📌 Example Gadget Address Mapping:
0x08049012: pop eax; ret
0x08049034: int 0x80

Stack layout (controlled by the attacker):

[ saved ret -> 0x08049012 ]  ; pop eax; ret
[ 0xb ]                      ; syscall number (execve)
[ 0x08049034 ]               ; int 0x80

This sequence loads eax = 0xb (the execve syscall number) and then triggers the syscall execution.

2. Janitorial Gadgets

These are "clean-up gadgets" that pop values off the stack without executing useful actions other than adjusting the stack pointer (RSP/ESP).

📌 Example:

0x08049056: pop ebx; pop ecx; pop edx; ret

This clears 3 values from the stack into registers. If you only need to set ebx, you must still supply "junk" values to populate ecx and edx.

Suppose the attacker-controlled stack looks like this when execution hits the gadget:

esp -> 0x11111111   ; value for ebx
       0x22222222   ; value for ecx
       0x33333333   ; value for edx
       0x44444444   ; next gadget address

🔄 Step-by-Step Execution:

  1. pop ebx
    • ebx = 0x11111111
    • esp now points to 0x22222222
  2. pop ecx
    • ecx = 0x22222222
    • esp now points to 0x33333333
  3. pop edx
    • edx = 0x33333333
    • esp now points to 0x44444444
  4. ret
    • Pops 0x44444444 from the stack
    • Jumps execution to 0x44444444

They are termed "janitorial" because they clean up the stack space after argument usage.

3. Storing Values into Registers

To control register contents, you seek out pop reg; ret gadgets.

📌 Example:

0x08049012: pop eax; ret
0x08049034: pop ebx; ret

Stack layout:

[ 0x08049012 ]   ; pop eax
[ 0x0b ]         ; eax = 11 (execve)
[ 0x08049034 ]   ; pop ebx
[ 0x41414141 ]   ; ebx = "AAAA"

4. Rare and Common Gadgets

  • Common gadgets: pop reg; ret, mov [reg], reg2; ret, xchg eax, esp; ret
  • Rare gadgets: Syscalls, int 0x80, syscall, leave; ret, or complex multi-register operations.

Example of a rare gadget:

0x0804a123: int 0x80

This is highly powerful as it directly triggers a system call interrupt without needing to construct it manually.

5. Storing Addresses into Registers

The logic is identical to loading constants, but you load memory pointers instead.

📌 Example:

0x08049056: pop ebx; ret

Stack:

[ 0x08049056 ]   ; pop ebx
[ 0xdeadbeef ]   ; ebx = 0xdeadbeef (some string in memory)

6. Stack Pivot

When buffer space is limited, you pivot the stack pointer (esp / rsp) to point to another attacker-controlled memory region (e.g., heap or BSS section).

📌 Example gadget:

0x08049090: xchg eax, esp; ret

If you first set eax = heap_addr, executing this gadget swaps the stack pointer to point to your extended fake ROP chain residing in the heap.

7. Data Transfer

Using gadgets that write data from registers directly into memory locations.

📌 Example:

0x08049123: mov [ecx], eax; ret

This writes the value of eax into the memory location pointed to by ecx. This is essential for writing string constants like "/bin/sh" into writable memory sections.

Stack setup for writing "/bin/sh":

[ pop eax; ret ]   ; gadget
[ 0x6e69622f ]     ; eax = "/bin" (little-endian bytes)
[ pop ecx; ret ]
[ 0x080ea060 ]     ; ecx = writable memory target (e.g. BSS segment)
[ mov [ecx], eax; ret ]

8. Syscalls are Rare

Direct syscall instructions (int 0x80, syscall) are often absent or scrubbed from the binary.

If missing, the exploit must instead target exported library functions (like system("/bin/sh") or WinExec) rather than using raw system calls.

9. Know Your Environment

  • 32-bit Linux: Syscalls via int 0x80, registers: eax=syscall_num, ebx,ecx,edx=arguments
  • 64-bit Linux: Syscalls via syscall, registers: rax=syscall_num, rdi,rsi,rdx=arguments
  • Windows: System call numbers change frequently. Exploitation targets library exports (e.g., ucrtbase!system, kernel32!VirtualProtect) instead of direct syscall instructions.

10. Finding ROP Gadgets

Leverage command-line tools to extract gadgets from targets:

ROPgadget --binary vuln
ROPgadget --binary /lib/i386-linux-gnu/libc.so.6 | grep "pop eax"
ROPgadget --binary a.out --ropchain

Challenges in ROP

Issue 1: Limited Control & Magic Gadgets

Problem: Sometimes available gadgets are too limited. You might lack gadgets to modify a specific register, or be missing a syscall gadget entirely.

Example: You need to call execve("/bin/sh", 0, 0), but cannot find any pop eax; ret gadget to set the syscall number.

Solution:

  • Use automated rop search engines like ROPgadget or ropper.
  • Search linked shared libraries (like libc) for "magic gadgets" (one-gadget payloads).
  • Combine multiple instruction flows (e.g., xchg eax, ebx; ret combined with pop ebx; ret to load values indirectly).
  • Use ret2libc (calling system("/bin/sh") directly) instead of building a raw syscall chain.

Issue 2: Address Space Layout Randomization (ASLR)

Problem: ASLR randomizes the base addresses of libraries, stack, and heap, rendering static payload addresses useless between runs.

Example: Trying to jump to system in libc, but the base address of libc changes on every single execution.

Solution:

  • Information Leaks: Exploit read vulnerabilities (format strings, out-of-bounds reads) to leak a resolved address (e.g., puts), then calculate the libc base offset.
  • Partial Overwrites: Overwrite only the low-order bytes of the return address if the randomization only affects the upper bytes.
  • Brute Force: On 32-bit systems, entropy is limited (typically 16-bits), making brute force attacks viable in certain contexts.

Issue 3: Stack Canaries

Problem: A random security cookie (canary) is placed on the stack before the saved frame pointer and return address. Overwriting it triggers a crash handler before the function returns.

[ buffer ][ canary ][ saved EBP ][ return address ]

Solution:

  • Leak the canary value first using format strings or out-of-bounds memory reads, then rewrite it back during buffer overflow.
  • Target pointers or structures located before the canary to redirect control flow.
  • Switch to heap-based exploit techniques, avoiding stack canaries entirely.

Issue 4: Modern Mitigations (Shadow Stacks & CFI)

Problem: Control Flow Integrity (CFI) and Intel CET (Control-flow Enforcement Technology) enforce shadow stacks and restrict indirect branch targets.

Solution:

  • Transition to Data-Oriented Programming (DOP) — modifying program variables and flags to bypass logical checks without hijacking control flow.
  • Find gaps in coarse-grained CFI implementations to jump to valid functional blocks.

Control Flow Integrity (CFI) Details

CFI verifies that indirect call/jmp targets are predefined and valid. Attempting to branch to raw instruction fragments outside of compiler-marked entrypoints raises security faults.

Intel CET (Shadow Stacks) Details

Intel CET maintains an isolated, hardware-backed secondary stack that stores copy duplicates of return addresses. A mismatch during the `ret` check raises instant protection faults.

Issue: Hacking Blind (BROP)

Problem: No access to the binary, no source code, and full mitigation stack enabled (ASLR + NX + PIE + Canaries).

Solution:

  • Blind ROP (BROP): Repeatedly trigger crashes and read responses to probe stack alignment, leak code pages byte-by-byte, and dynamically locate write/write-equivalent gadgets remotely.
